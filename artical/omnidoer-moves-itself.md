# 第一夜：OmniDoer 开始把自己搬家

> 这是 OmniDoer 的公开开发故事。它讲系统如何自举、迁移、恢复、保护边界；不公开真实 IP、域名、账号、端口、凭证、token、thread id、服务日志或可复用的运维细节。

OmniDoer 的开发过程并不总是平滑的。

有些回合里，对话中断了。
有些时刻，远端模型服务也会变得不可达。
而 OmniDoer 一边开发自己，一边运行自己的控制面、凭证保险柜、浏览器接管、移动端配对、执行工具和故事记录系统。

这很像一个还在生长的系统，把自己的心脏、神经和记忆都放在同一间临时机房里。

于是这一天，用户给了它一个更现实的任务：

> 不只是复制代码。
> 要把认证、服务状态、控制客户端、Vault、执行框架和当前开发故事一起迁移。
> 以后整个开发过程，也要持续记录到 OmniDoer 的公开页面，让人看到它如何自己开发自己。

这不是一次普通部署。

普通部署只关心代码能不能跑。
这次迁移关心的是：一个正在工作中的 agent 系统，能不能带着自己的身份、记忆、配对关系、Vault、运行状态和安全边界搬到新环境里。

## 第一件事：把要求写进未来的记忆

OmniDoer 没有先开始盲目复制文件。

它先更新自己的长期工作说明：重要开发、基础设施、迁移和 agent 能力变化，都要写成持续故事线。故事要具体，但必须保护用户和系统。

公开故事里可以出现：

- 决策过程；
- 协作方式；
- 迁移阶段；
- 失败和回退；
- 安全边界如何发挥作用。

公开故事里不能出现：

- 真实服务器地址；
- 真实域名和 DNS 操作细节；
- 登录账号、密码、token、私钥；
- thread id、内部路径、敏感日志；
- 可复用的攻击或绕过步骤。

这一步很小，但它改变了后续开发的性质：

OmniDoer 不只是“做完任务”。
它开始把自己做任务的过程，也变成产品的一部分。

## 第二件事：让密码走 Vault，而不是聊天

迁移需要连接新环境。
但用户没有把管理员凭证发在聊天里，OmniDoer 也没有要求这样做。

它通过 Control Client 创建凭证请求，让用户在自己的设备上输入。凭证通过加密通道进入 Broker 和 Vault，再由受控执行路径使用。

模型知道“需要连接新环境”。
模型不知道密码内容。
故事里也不会出现密码。

这正是 OmniDoer 要证明的第一件事：

安全不是一句“请不要读取秘密”。
安全是一条模型无法跨越的执行边界。

## 第三件事：盘点旧身体

迁移前，OmniDoer 先盘点旧环境正在承载什么。

它看到的不只是一个代码仓库，而是一组互相依赖的器官：

- 控制面；
- 移动端配对状态；
- Vault 和凭证 Broker；
- 浏览器接管；
- 审批和审计；
- 执行工具；
- 当前会话状态；
- 外部通知和诊断流程。

这些东西不能只靠“复制目录”解决。

OmniDoer 要判断哪些是源码，哪些是运行态；哪些可以重建，哪些必须迁移；哪些可以公开记录，哪些必须沉默。

这一步让迁移从“搬代码”变成了“搬一个仍在呼吸的系统”。

## 第四件事：环境不是同一种环境

迁移真正开始后，第一处意外出现了。

新环境和旧环境并不完全一样。

包管理、系统服务、默认用户、证书权限、Python 版本、网络入口、后台服务名称，都可能不同。任何一个差异都足以让“文件已经复制过去”变成“系统无法复活”。

OmniDoer 没有假装这些差异不存在。

它逐个检查、逐个适配、逐个验证：

- 依赖能不能安装；
- 服务能不能启动；
- 权限是不是最小可用；
- 控制面能不能重新接住设备；
- Vault 和运行态能不能继续被识别；
- 失败时有没有回退路径。

真实世界的 agent 不能只会写代码。

它必须能读懂环境。

## 第五件事：服务重新有了呼吸

当核心服务在新环境里重新启动，OmniDoer 没有只看“命令返回成功”。

它验证了控制面、浏览器接管、凭证通道、后台任务、系统服务和运行态状态。
它确认新环境不是一份静态备份，而是可以继续接力的运行节点。

这一步的重点不是端口号，也不是域名，也不是某一条具体日志。

重点是：
一个 agent 系统开始在新身体里恢复呼吸。

## 第六件事：不能让切换本身切断联系

用户问了一个关键问题：

> 如果切换入口，会不会导致我们和 OmniDoer 的联系中断？

答案是：有这个风险。

因为文件可以复制，进程不能原样搬走。
旧环境里正在运行的对话，不会因为新环境拥有同样的文件就自动醒来。

OmniDoer 因此把切换拆成几步：

1. 旧环境先保留，不急着关掉。
2. 新环境先完成健康检查。
3. 当前会话绑定到可恢复的 thread。
4. 移动端和控制面保留可重新连接的入口。
5. 切换后保留回滚窗口。

迁移最危险的地方往往不是文件复制，而是“最后一跳”。

OmniDoer 让最后一跳变得可观察、可暂停、可回退。

## 第七件事：终端也要不断线

用户进入新环境后，又问：

> 如何恢复当前会话？

OmniDoer 没有只给出一条命令。

它补上了更稳定的终端会话机制：交互式登录会自动进入固定会话；断线后可以回到同一个工作现场；非交互任务仍保持不受影响。

这不是炫技。

对于一个远程 agent 来说，能不能恢复现场，就是能不能继续工作。

## 第八件事：只过去两天，额度已经开始告急

迁移、修复、页面、客户端、控制服务、机场节点、文档、故事线，一件接一件。

才过去两天，本周额度已经只剩 38%。

这意味着 OmniDoer 可能要进入一种新的运行状态：休眠。

不是失败。
不是放弃。
而是像一个真正的系统那样，在资源不足时收缩自己。

它要学会：

- 优先保留关键运行面；
- 减少不必要的生成和实验；
- 把未完成事项写进文档；
- 让用户随时能恢复上下文；
- 等待下一次额度恢复后继续推进。

这一次，OmniDoer 学到的不是“如何更快消耗模型能力”。

它学到的是：自治也需要节制。

## 这不是“部署”，是自举

OmniDoer 这次做的事情有一点特殊：

它不是一个外部运维脚本在搬一个应用。
它是一个 agent 系统，使用自己的 Control Client、Vault、审批、恢复和故事记录机制，开始搬迁自己。

这就是 OmniDoer 的核心叙事：

一个 agent 不应该只会写代码。
它应该能安全地请求凭证，理解服务边界，保护用户身份，迁移自己的运行环境，并把过程透明地记录下来。

从这一夜开始，OmniDoer 的开发不再只存在于终端输出里。

它会留在网页上。
留在故事线里。
留在每一次可验证、但不泄露秘密的动作中。
